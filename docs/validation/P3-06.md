# P3-06 — Warriors, warrior structures with upgrade tree, night escalation

## What changed

The professional night tier layered on the P3-05 light-band combat: Dark-capable
**warriors** promoted from villagers at a **lodge**, data-driven **structure upgrades**,
a year-long **night-escalation curve**, and a nightly **dark objective** only a
dark-capable unit can solve. All balance lives in the **`CombatConfig`** ScriptableObject
(Resources path `CombatConfig`, coded defaults — no asset file required): a warrior
section (base stats, lodge capacity, patrol radius, watchtower vision/support), the
`warriorUpgradeTiers` tree (costs in Phase 3 tools/scrap_iron/coal + stat deltas), and a
night-escalation section (base/growth/season multipliers, set-piece cadence, wave cap).

- **`Runtime/Combat/WarriorAgent.cs`** — a Dark-capable unit: musters at dusk, patrols/
  engages out in the dark, solves the nightly objective, returns at dawn. Fights through
  `LightBandCombatResolver` as a **Friendly** (NOT band-exempt — only the beast is), so
  its Dark strikes are debuffed and it pays the Dark sanity drain, reduced only by its
  trained `DarkSanityDrainFraction` (upgrades lower it).
- **`Runtime/Combat/WarriorUpgrades.cs`** — the pure upgrade tree: `WarriorUpgradeTier`
  data, `StatsAtTier` (cumulative deltas from the config base), and `TryPurchaseTier`
  (the single ledger debit).
- **`Runtime/Combat/WarriorStructure.cs`** — the lodge (recruit villagers → warriors with
  population conservation, own the upgrade tree, `TryUpgrade`) and the watchtower (ranged
  support + the vision that arms the objective marker).
- **`Runtime/Nightmares/NightEscalationSystem.cs`** — static curve math (`WaveBudget`,
  `IsSetPieceNight`, `WaveMonsterCount`) + the nightly dark-objective lifecycle
  (`BeginNight`/`ResolveNight`, `DarkObjectiveMarker`, `DarkObjectiveSolved/Failed`).
- **`Runtime/Nightmares/DarkObjectiveGenerator.cs`** — seeded generator: same kind +
  location for a given seed/night, location always classifies Dark at generation time.
- **Modified**: `NightmareDirector` (Phase 3 branch behind `phase3NightsEnabled` —
  spawns the escalation wave + drives the objective), `BuildingCatalog` (+`warrior_lodge_t1`,
  `watchtower_t1`; +`FunctionKind.WarriorLodge/Watchtower`), `Building.Construct`
  (attaches `WarriorStructure`), `CombatDebugPanel` (F6: escalation + warrior roster),
  `PrototypeConfig` (`phase3NightsEnabled`), `PrototypeSceneBuilder` (lodge + watchtower +
  `NightEscalationSystem`).
- **Assets**: `blender/asset_specs/buildings/warrior_lodge_t1.json`, `watchtower_t1.json`
  + generated GLB / previews / metadata.

Public API downstream tasks consume: `WarriorAgent` (`State`, `Stats`, `Sanity`,
`Active`), `WarriorStructure` (`Recruit`, `TryUpgrade`, `AppliedTierCount`, `Role`),
`NightEscalationSystem.WaveMonsterCount`/`IsSetPieceNight`/`BeginNight`/`ResolveNight` and
`DarkObjectiveSolved`/`Failed` (P3-09 laws gate night labour/warrior doctrine via the
recruit/upgrade path and the `phase3NightsEnabled` toggle; P3-11 threat-source spawns read
the escalation wave budget/set-piece flag). The `night_escalation`, `dark_objective`,
`warrior_*` log records feed the morning report.

## Technical verification

Run in the local macOS Unity editor with MCP for Unity (the compile/test authority; CI
never compiles Unity — see `docs/VERIFICATION_STATUS.md`):

1. Start / connect the pinned editor + MCP bridge:
   - `tools/restart_unity_mcp.sh` (fresh), or
   - `UNITY_MCP_UV_OFFLINE=1 tools/run_unity_mcp_gate.sh --no-restart` when already
     connected.
2. Full gate: `tools/run_unity_mcp_gate.sh` (Unity gate + EditMode + PlayMode + console
   check). Everything must be green with **no console errors**.

New test fixtures for this task:

- EditMode: `Abbey.Tests.EditMode.WarriorUpgradeTests` — **14 tests** (tier-0 stats are the
  config base; tier deltas apply cumulatively; the tree clamps to its length; `TryUpgrade`
  debits the ledger and raises housed-warrior stats; an unaffordable upgrade leaves the
  ledger untouched; recruiting promotes a villager and conserves population; a full lodge
  recruits no more; the wave budget is monotonic within a season; season multipliers step
  up Spring→Summer→Autumn→Winter; set-piece nights spike the wave; the wave count clamps to
  the config cap; the objective generator reproduces its sequence for a fixed seed; the
  objective always classifies Dark; successive nights differ).
- PlayMode: `Abbey.Tests.PlayMode.NightEscalationPlayModeTests` — **4 tests** (a warrior
  musters and engages a monster in the Dark band, wins per stat config, and loses sanity —
  it is not band-exempt; a warrior solves the dark objective and it clears with no
  consequence at dawn; an unsolved objective fires `DarkObjectiveFailed` and its downed
  villager is lost to the dark; a set-piece night spawns the larger wave the curve
  prescribes).

Expected totals after this task: EditMode **224 + 14 = 238**, PlayMode **52 + 4 = 56**
(baseline after P3-05 was 224 / 52 — see `docs/validation/P3-05.md`). A different count
means a regression or a merge changed the set — investigate before trusting the gate.

Container check (no Unity): `./tools/check_all.sh` — must pass. Every Unity step SKIPs

## Visual verification

1. `Tools → Abbey → Build Prototype Scene`, then press Play. The camp now also places a
   **WarriorLodge** (dark edge, `-8,0,6`) and a **Watchtower** (`8,0,-4`), and a
   `NightEscalationSystem` on `WorldSystems`.
2. Press **F6** (`CombatDebugPanel`). Below the band multipliers and home rows you now see:
   - **Night escalation**: tonight's `night`, `season`, `standard`/`SET-PIECE`, the
     `wave budget` and the derived `monsters`, and the `objective kind [state] band`.
   - **Warriors**: the lodge row (`tier x/y`, `roster n/cap`), the watchtower row, the
     count of warriors afield, and per-warrior `[State] band hp san`.
3. Enable Phase 3 nights: set `phase3NightsEnabled = true` on the `PrototypeConfig`
   (Resources asset, or the coded default) so the director runs the escalation wave +
   objective. Recruit warriors at the lodge via the debug path / `WarriorStructure.Recruit`,
   then fast-forward to night.
   **Correct (warriors hold the dark):**
   - warrior units **visibly leave the lit ring** and move out into the dark to muster /
     engage — the contrast with P3-05 settlers, who stay home behind flared windows;
   - the **dark-objective marker sits outside the light gizmos** (`band=Dark` on F6), and
     **clears when a warrior reaches it** (row flips to `SOLVED`, `DarkObjectiveSolved`
     fires); an unsolved objective at dawn shows `failed … consequence=…` in the log and a
     villager goes Missing;
   - engaged warriors' `san` on F6 **ticks down** while they fight in Dark (they are not
     band-exempt), but they win where settlers would be overrun.
   **Correct (escalation):** advance several nights (and into Autumn/Winter): the F6
   `wave budget`/`monsters` **grow**, and every Nth night reads **SET-PIECE** with a
   noticeably larger wave on the `NightmareDebugPanel` monster count.
   **Failure:** warriors stay inside the light, a razed/settler-only night with no warriors
   afield, the objective marker sits inside a Safe gizmo, a warrior suffers no Dark sanity
   drain (only the beast is exempt), or the wave budget is flat across nights/seasons.
4. Cross-check balance against data, not code: open `CombatConfig` (or the coded defaults
   in `Runtime/Combat/CombatConfig.cs`) and confirm the warrior base stats,
   `warriorUpgradeTiers` costs/deltas, the season multipliers and `escalationSetPieceEveryNNights`
   match what the F6 panel shows.
