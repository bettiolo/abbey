# P3-14 — Spring-ship win, four chapters and end summary

## What changed

The Phase 3 campaign close. A new opt-in flag **`phase3CampaignEnabled`** on
**`GameSessionConfig`** (Resources path `GameSessionConfig`, coded defaults — no asset
required) turns the run into the four-chapter campaign whose goal is to rebuild a ship and
sail out the following spring. All balance (chapter names/flavour, launch year, manifest
thresholds, ship id) lives on that config. Four new/extended systems, all under
`Runtime/Session/` unless noted:

- **`ChapterSystem`** (singleton, `[ExecuteAlways]`) — the four narrative chapters keyed to
  the calendar's four seasons: **The Wreck** (spring), **The Meadow** (summer), **The Long
  Rain** (autumn), **The First White Night** (winter). A chapter transition logs a
  `chapter` record with morning-report flavour (campaign mode only) and raises
  `ChapterChanged`.
- **`SpringShipScenario`** (singleton) — tracks the three-part launch manifest and, at the
  spring launch window (year `springLaunchYear`, default 2), sails the ship when the
  manifest is complete. Parts, each evaluated independently: **settlers** (willing sailors =
  P3-13 trust volunteers + spring departure intents ≥ `manifestSettlers`), **provisions**
  (ledger thresholds in `manifestProvisions`; grain folds into food at
  `EconomyConfig.GrainToFood` first), **hull/rigging** (the `spring_ship_t1` construction
  site completes — its cost includes **wool woven into sailcloth**). A missing manifest at
  the window is **not** a loss: it logs `ship_did_not_sail` and the year rolls on.
- **`CampaignOutcome`** — the serialized Phase 4 carryover: result, chapter reached, manifest
  state, who-sails/who-stays roster, key choices (hound path, abbey form, trust tier, laws,
  deaths, POIs) and the chronicle. Written to `Application.persistentDataPath/campaign_outcome.json`
  on the win.
- **`Runtime/Reports/EndSummary.cs`** — the data-driven end-of-run chronicle in the P2-07
  storybook voice, reduced purely from the `GameEventLog` (chapters, laws, deaths, hound
  path, abbey form, threat history, launch).

`GameSession` gained the `ShipSailed` outcome + `WinterCollapse` loss reason: in campaign
mode the White Night is chapter 4's climax (not the run's end), the campaign win is latched
by `SpringShipScenario.ReportShipSailed`, and a winter wipe before spring is its own loss.
`GameOutcomePanel` (F6) renders the campaign chronicle + roster. The **`DebugOverlay` (F1)**
gained a `Campaign (P3-14)` section: current chapter (n/4) + year/season, the three manifest
parts (settlers/provisions/hull) with a COMPLETE/incomplete verdict, and launch-window/sailed
state. No new debug key was taken.

Added: `Runtime/Session/ChapterSystem.cs`, `Runtime/Session/SpringShipScenario.cs`,

`blender/asset_specs/buildings/spring_ship_t1.json`,

reconstruction site at the wreck).

## Visual verification

1. Preview `spring_ship_t1` (once generated): `blender/generated/previews/spring_ship_t1_*.png`
   — a single-masted coastal ship on a launch cradle, clinker hull + furled canvas sail.
2. Enable the campaign: set **`phase3CampaignEnabled = true`** on a
   `Resources/GameSessionConfig` asset (create one via `Assets → Create → Abbey/Game Session
   Config` if none exists), then `Tools → Abbey → Build Prototype Scene` and press Play.
   `WorldSystems` now also holds a `ChapterSystem`; a `SpringShipScenario` and a
   `SpringShipSite` (the ship greybox) sit near the wreck on the SW beach.
3. Press **`F1`** (DebugOverlay). The new **Campaign (P3-14)** section shows
   `chapter 1/4: The Wreck  year 1 — Spring …` and the manifest line
   `settlers -- (0/3)  provisions --  hull --  => incomplete`. **Correct** as the year runs:
   the chapter banner advances **The Wreck → The Meadow → The Long Rain → The First White
   Night** at each season turn (also logged as `chapter` records), and the night length grows
   toward winter.
4. Use debug grants to complete the manifest: deliver wood/tools/wool to the ship site (the
   greybox grows through the build), grant the provisions (food/candles/tools), and let
   trust rise so newcomers volunteer (or record spring departures). The overlay's manifest
   line flips each part to `OK` and finally `=> COMPLETE`.
5. Fast-forward to the following spring (year 2). **Correct:** the launch window opens
   (`launch window: OPEN`), the ship sails (`spring_ship launched result=ShipSailed …`),
   `GameOutcomePanel` (F6) auto-shows titled **The Abbey at World's End** with the
   `ShipSailed` headline, the manifest ✓/✓/✓, the who-sails/who-stays roster, and the
   end-summary chronicle. **Spot-check 3 facts** against the event log (F1 log / console):
   the chronicle's named law, hound path and death count must match the real `decree` /
   `hound_evolved` / `villager_died` records of the run. **Broken** = the chapter banner never
   advances, the manifest never completes, the ship sails with an empty manifest, or the
   chronicle names facts that never happened.
6. Failure branches: with an incomplete manifest at spring the overlay/log shows
   `ship_did_not_sail` and the run continues (no end screen). A settlement wiped in winter
   ends with the **WinterCollapse** loss on `GameOutcomePanel`.
7. Cross-check balance against data, not code: open `GameSessionConfig` and confirm
   `chapterNames`, `springLaunchYear`, `manifestSettlers`, `manifestProvisions` and
   `shipBuildingId` match what the overlay and the `chapter` / `spring_ship` log records show.
