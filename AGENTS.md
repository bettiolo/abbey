# Agent operating rules

These rules bind any LLM agent (and any human acting like one) working in this repo.

## The loop

Work through repeatable commands, in this order, for every asset or system change:

```
generate asset spec → generate Blender script → run Blender headless → export GLB
→ render previews → validate asset → import into Unity → run Unity import validation
→ run Unity tests → capture screenshot → fix issues → commit
```

Never create assets ad hoc. Every asset starts as a spec file in `blender/asset_specs/`.

## Always

- Make small changes.
- Run tests after gameplay changes (`./tools/run_unity_tests.sh` or CI).
- Generate previews after asset changes (`python3 tools/run_blender_asset_pipeline.py`).
- Never hand-edit generated files (`blender/generated/`, `unity/Assets/Generated/`) unless
  explicitly told.
- Keep design constants in data files or ScriptableObjects — never hard-code balance values
  inside MonoBehaviours.
- Prefer deterministic systems (seeded RNG, fixed tick).
- Add debug overlays for every hidden system.
- Use placeholder assets before polishing.
- Commit only working milestones.
- Record every generated asset in metadata (`blender/generated/metadata/*.meta.json`).

## Never

- Do not build a huge world first.
- Do not start with final art.
- Do not create bespoke one-off materials — use only the shared material library
  (`blender/scripts/abbey_materials.py`).
- Do not use Unity scene files as the source of truth for core game logic. Scenes are
  generated by editor bootstrap code (`Tools → Abbey → Build Prototype Scene`).
- Do not hide balance values inside MonoBehaviours.
- Do not create final animations before the core loop works.
- Do not rely on generated 3D assets without preview validation.
- Do not mark an asset complete unless `validate_asset` passes every check.

## Definition of done

A task is not complete until `./tools/check_all.sh` passes **or the failure is clearly
documented** in the commit message / report.

## Environment notes

- Blender may be available as the `blender` binary or as the `bpy` PyPI module; all pipeline
  scripts must work with either (see `tools/run_blender_asset_pipeline.py`).
- The Unity editor is typically NOT available in the agent container. Unity compilation and
  test execution happen in CI (GameCI). Author Unity code so that PlayMode tests construct
  their world programmatically and never depend on hand-authored scene files.
- Generated `.blend` files are build artifacts and are gitignored. Source of truth is the
  spec + generator script. `.glb`, previews, and metadata ARE committed so Unity CI can
  import them without running Blender.

## Human review gates

1. **Art bible** — camera angle, scale, palette, day/night contrast, first building kit,
   hound silhouette.
2. **First playable night** — is day charming, dusk tense, night scary? Does light feel
   like territory? Is the hero necessary?
3. **Beast bond** — does the hound feel alive, distinct from a normal unit, emotionally
   meaningful?
4. **Vertical slice** — is the loop fun for 20 minutes? Real game, not an AI asset collage?

Do not advance a phase past its gate without human approval.
